# Iceberg — 3 примера (Lever 11 / Hemingway)

> [!info] Контекст
> Iceberg — Pass 2 Phase RELY в `humanize-writer` и `humanize-editor`. Применяет Hemingway's iceberg theory: оставить пробелы, которые читатель заполнит сам. «If a writer of prose knows enough of what he is writing about he may omit things that he knows and the reader... will have a feeling of those things as strongly as though the writer had stated them.»

> [!warning] Условие Хемингуэя
> Iceberg работает только если автор **знает** то, что не говорит. Если не знает — это hollow place, не gravitas. Когда не уверены — пишите явно.

Каждый пример содержит:
- AI-typical текст (over-explanation)
- Iceberg версию
- Что подразумевается (но не сказано)
- Reader-fill test (что читатель сам поймёт)

---

## Пример 1: Архитектурное решение

### ❌ До (AI-typical, over-explanation)

> Мы выбрали PostgreSQL в качестве основного хранилища данных, потому что PostgreSQL предоставляет ACID-гарантии, поддерживает сложные запросы с JOIN, имеет богатую экосистему расширений (PostGIS, pg_trgm, pg_stat_statements), и проверен в продакшене многими крупными компаниями. Альтернативы в виде MongoDB мы рассмотрели и отклонили из-за отсутствия ACID, а Cassandra — из-за eventual consistency. MySQL мы также рассматривали, но PostgreSQL показал лучшую производительность на наших benchmark'ах.

### ✅ После (Iceberg, +43% ёмкость)

> **PostgreSQL.** ACID, JOIN'ы, проверен в продакшене. MongoDB и Cassandra не подошли — eventual consistency ломает наши транзакции. MySQL медленнее на наших запросах.

### Что подразумевается (но не сказано)

- **«ACID-гарантии»** — читатель знает, что ACID = Atomicity, Consistency, Isolation, Durability. Не нужно объяснять.
- **«JOIN, экосистема расширений (PostGIS, pg_trgm, pg_stat_statements)»** — список расширений это filler. Достаточно «JOIN'ы».
- **«проверен в продакшене многими крупными компаниями»** — social proof через перечисление компаний (Stripe, Instagram, etc.) не добавляет решения.
- **«benchmark'ах»** — детали benchmark'ов избыточны. Достаточно «на наших запросах».

### Reader-fill test

Вопрос: удалили последнее предложение про «benchmark'ах». Потерял ли читатель что-то критичное?
- ❌ Если это академическая статья — да, нужно привести цифры.
- ✅ Если это решение для команды — нет, читатель уже понял, что PostgreSQL выбран.

**Решение:** iceberg работает для внутреннего обсуждения. Для статьи — добавить конкретные цифры benchmark'а.

## Пример 2: Решение bug'а

### ❌ До (AI-typical, over-explanation)

> На прошлой неделе в нашей системе был обнаружен серьёзный баг, который приводил к утечкам памяти в production-окружении. После тщательного расследования мы выяснили, что причиной была некорректная работа с контекстными переменными в middleware-слое нашего приложения. Мы исправили баг путём добавления явного вызова функции очистки контекста после завершения обработки запроса. Дополнительно мы добавили интеграционные тесты, которые проверяют отсутствие утечек памяти при обработке 1000 последовательных запросов.

### ✅ После (Iceberg, +62% ёмкость)

> На прошлой неделе пофиксили memory leak в middleware. Причина: контекст не очищался после запроса. Добавили integration test на 1000 последовательных запросов.

### Что подразумевается (но не сказано)

- **«серьёзный баг»** — если это был несерьёзный, не постили бы. Удаление эпитета не теряет смысла.
- **«production-окружении»** — где ещё может быть утечка? Только в prod.
- **«тщательного расследования»** — это filler. Расследование очевидно.
- **«явного вызова функции очистки контекста»** — деталь реализации, не нужна в summary.
- **«после завершения обработки запроса»** — это объяснение `defer context.Cancel()` в Go. Очевидно для Go-разработчика.

### Reader-fill test

Вопрос: читатель — Go-разработчик в команде. Что он поймёт без объяснений?
- ✅ Что был memory leak → понятно из заголовка.
- ✅ Где — middleware → понятно.
- ✅ Как починили — контекст не очищался → понятно без «явного вызова функции очистки».
- ✅ Тест есть → достаточно.

**Решение:** iceberg работает для команды. Для внешнего поста (например, на Habr) — добавить немного больше деталей.

## Пример 3: Code review feedback

### ❌ До (AI-typical, over-explanation)

> Спасибо за ваш pull request! Я провёл тщательный обзор предложенных изменений и обнаружил несколько моментов, которые требуют доработки. Во-первых, в функции calculateTotal() отсутствует обработка случая, когда массив items пуст, что может привести к ошибке деления на ноль. Во-вторых, рекомендуется добавить unit-тесты для edge cases, чтобы покрыть негативные сценарии. В-третьих, именование переменной tmp1 не соответствует code style проекта, рекомендуется использовать более описательное имя. В остальном код выглядит хорошо, продолжайте в том же духе!

### ✅ После (Iceberg, +50% ёмкости)

> Несколько правок:
>
> 1. `calculateTotal` падает на пустом массиве (division by zero). Добавь guard.
> 2. Тесты для edge cases нужны.
> 3. `tmp1` → описательное имя.
>
> В остальном ок.

### Что подразумевается (но не сказано)

- **«Спасибо за ваш pull request!»** — вежливость без информации.
- **«Я провёл тщательный обзор»** — подразумевается, что вы рецензент, иначе не пишете review.
- **«обнаружил несколько моментов, которые требуют доработки»** — стандартная формулировка, ничего не добавляет.
- **«в функции calculateTotal()»** — код-ревью, очевидно, что речь о коде.
- **«может привести к ошибке деления на ноль»** — разработчик и так знает, что деление на ноль это баг.
- **«соответствует code style проекта, рекомендуется использовать более описательное имя»** — само review замечание достаточно.

### Reader-fill test

Вопрос: читатель — автор PR, senior developer. Что он поймёт?
- ✅ Что нужно править → понятно из bullet-списка.
- ✅ Что именно — guard для division by zero → понятно.
- ✅ Что `tmp1` плохое имя → понятно, это code smell.
- ❌ Что «в остальном ок» — это approval. Если убрать, может показаться что весь PR rejected.

**Решение:** iceberg работает для routine reviews. Для critical PR — можно оставить «в остальном ок» как marker.

## Принцип iceberg: когда работает, когда нет

### ✅ Iceberg работает

- **Email коллеге** — общий контекст, не нужно объяснять.
- **RFC внутри команды** — все знают архитектуру.
- **Комментарий в PR** — рецензент и автор в одном контексте.
- **Пресс-релиз для отрасли** — читатели разделяют профессиональный жаргон.
- **Status update для знакомых** — контекст очевиден.
- **Художественная проза** — читатель сам достраивает.

### ❌ Iceberg не работает

- **Onboarding нового сотрудника** — нет контекста, нужно объяснить.
- **Tutorial для новичков в теме** — общий контекст = 0.
- **Юридический документ** — пропущенное слово может изменить смысл.
- **API reference** — completeness обязательна.
- **RFC для другой команды** — у них другой контекст.
- **Пресс-релиз для широкой публики** — general audience, общий контекст минимальный.
- **Научная статья** — peer review требует explicit claims.

## Эвристики iceberg (Lever 11)

### Reader-fill test

Удалите абзац. Читатель может продолжить мысль без него? Если да — удалите.

### Distinctive-word test

Если в предложении нет ни одного слова, добавляющего информацию, которой не было в предыдущем, — удалите его.

### Stop at the turn

Если в абзаце есть «поворотный момент» (решение, открытие, конфликт) — остановитесь на нём. LLM продолжает объяснять, что это значит. Хороший автор ставит точку.

### Конкретные правила

1. **«серьёзный», «критический», «важный»** — обычно эпитеты-filler, удаляйте.
2. **«тщательное расследование», «детальный анализ», «подробное изучение»** — adverbs-filler, удаляйте.
3. **«спасибо», «пожалуйста», «не стесняйтесь»** — в tech-тексте: удаляйте.
4. **«продолжайте в том же духе», «отличная работа»** — в code review: удаляйте или переформулируйте.
5. **Объяснения очевидного** («это функция, которая делает X») — удаляйте.
6. **Списки примеров в скобках** (`(Stripe, Datadog, PlanetScale)`) — оставьте один, остальные удаляйте.

## Reader-fill vs Missing-info различие

| | Underspecification (хорошо) | Missing-info (плохо) |
|---|---|---|
| Читатель | заполняет пробелы | не понимает контекст |
| Результат | «ага, понятно» | «подожди, что?» |
| Тест | можете продолжить без этого? | понимаете ли это без этого? |
| Пример | «PostgreSQL.» (без объяснения почему) | «Используем DB.» (без указания какой) |

**Принцип:** iceberg = пробел, который читатель заполняет. Missing info = пробел, который фрустрирует.
